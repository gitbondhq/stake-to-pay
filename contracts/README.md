# MPPEscrow Contracts

Solidity contracts for the MPP stake payment intent. `MPPEscrow` is a minimal,
auditable escrow that locks whitelisted ERC-20 tokens under a unique stake key.
A designated counterparty (or its delegates) can later refund or slash the
deposit.

This contract is designed to be **forked and extended** for your own
stake-to-pay application, not used as-is in production.

## Directory layout

```
contracts/
  src/
    IMPPEscrow.sol      Interface: structs, events, errors, function signatures
    MPPEscrow.sol        Reference implementation with lifecycle hooks
  test/
    MPPEscrow.t.sol      Foundry tests (unit + fuzz)
  script/
    DeployMPPEscrow.s.sol   Deployment script
```

## How the escrow works

### Lifecycle

1. **Create** -- A payer calls `createEscrow` after approving the contract to
   spend their tokens (`ERC20.approve`). Tokens transfer into the contract and
   an `Escrow` record is stored under the provided stake key.

2. **Refund** -- The counterparty (or a refund delegate) calls `refundEscrow`.
   Tokens return to the beneficiary. This is the happy path -- the payer
   behaved well and gets their collateral back.

3. **Slash** -- The counterparty (or a slash delegate) calls `slashEscrow`.
   Tokens transfer to the counterparty as a penalty.

Once resolved (refunded or slashed), an escrow is permanently inactive.

### Key concepts

| Concept             | Description                                                                                                                                                   |
| ------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Stake key**       | A unique `bytes32` identifier binding an escrow to a specific MPP challenge. Generated by the SDK during the 402 handshake.                                   |
| **Counterparty**    | The address authorized to resolve (refund/slash) the escrow. Typically the server operator's address.                                                         |
| **Beneficiary**     | The address that receives tokens on refund. Defaults to the payer if set to `address(0)`.                                                                     |
| **Delegates**       | Addresses the counterparty authorizes to call refund or slash on their behalf. Useful for separating hot-wallet operations from the counterparty key.         |
| **Token whitelist** | Only pre-approved ERC-20 tokens can be escrowed. This guards against fee-on-transfer, rebasing, and other non-standard token mechanics that break accounting. |

### Lifecycle hooks

`MPPEscrow` exposes three internal hooks you can override in your fork:

```solidity
function _afterDeposit(bytes32 key, address token, uint256 amount) internal;
function _beforeRefund(bytes32 key, address token, address beneficiary, uint256 amount) internal;
function _beforeSlash(bytes32 key, address token, address counterparty, uint256 amount) internal;
```

All are no-ops by default. Example uses:

- **`_afterDeposit`** -- Route locked tokens into a lending pool or yield vault.
- **`_beforeRefund`** -- Withdraw from the vault before disbursing, or deduct a
  service fee.
- **`_beforeSlash`** -- Take a platform cut before sending the remainder to the
  counterparty.

## Where to make changes

### Adding a new token

Pass the token address in the `WHITELISTED_TOKENS` env var at deploy time
(comma-separated). There is no admin function to add tokens after deployment --
this is intentional to keep the contract simple. If you need dynamic whitelisting,
add an `onlyOwner` mutator to the contract.

### Custom resolution logic

Implement `_beforeRefund` or `_beforeSlash` in your fork. For example, a
time-locked refund:

```solidity
contract TimedEscrow is MPPEscrow {
    uint256 public constant MIN_LOCK = 1 hours;

    constructor(address[] memory tokens) MPPEscrow(tokens) {}

    function _beforeRefund(bytes32 key, address, address, uint256) internal view {
        require(block.timestamp >= s_escrows[key].depositedAt + MIN_LOCK, "too early");
    }
}
```

### Yield on deposits

Override `_afterDeposit` to supply tokens to a lending protocol (e.g. Aave,
Compound), and `_beforeRefund` / `_beforeSlash` to withdraw before settlement.

### Access control

The base contract has no owner. If you need admin functions (pause, emergency
withdraw, dynamic whitelist), add OpenZeppelin's `Ownable` or `AccessControl`
and gate those functions accordingly.

## Extending token approval patterns

This kit ships with the simplest approval flow: the payer calls `ERC20.approve`
then `createEscrow` (two transactions). There are two common extensions you
could add to improve UX:

### ERC-2612 Permit

[ERC-2612](https://eips.ethereum.org/EIPS/eip-2612) lets the payer sign a
gasless approval off-chain. To support this:

1. Add a `PermitParams` struct to the interface (deadline, v, r, s).
2. Add a `createEscrowWithPermit` function that calls
   `IERC20Permit(token).permit(...)` before `safeTransferFrom`.
3. Import `IERC20Permit` from OpenZeppelin.
4. On the SDK side, build the ERC-2612 typed-data signature and call the new
   function in a single transaction.

This reduces escrow creation from two transactions to one, but only works with
tokens that implement ERC-2612 (most major stablecoins do).

**Security note:** ERC-2612 permit only signs the token spend (spender, amount,
deadline). It does not cover the escrow terms (key, counterparty, beneficiary).
This means a permit signature can be front-run -- an attacker who observes the
pending transaction could extract the permit signature and use it to create an
escrow with different terms. If this is a concern, use Permit2 with a witness
hash instead, which binds the approval to the full escrow parameters.

### Permit2

[Permit2](https://github.com/Uniswap/permit2) is Uniswap's universal approval
router. It works with any ERC-20 token (the payer approves the Permit2 contract
once, then signs per-spend permits). To support this:

1. Add a Permit2 integration that calls
   `IPermit2.permitTransferFrom(...)` with a witness hash encoding the full
   escrow terms (key, counterparty, beneficiary, amount).
2. This binds the approval to the exact escrow parameters, preventing relay
   attacks where a third party reuses the signature with different terms.
3. Import the Permit2 interface and deploy against the canonical Permit2
   contract on your target chain.

Permit2 with witness is strictly stronger than ERC-2612 for escrow use cases
because the signed message covers the escrow terms, not just the token spend.
This eliminates the front-running risk described above.

## Building and testing

Requires [Foundry](https://book.getfoundry.sh/getting-started/installation).

```bash
# Run tests
forge test

# Run tests with gas reporting
forge test --gas-report

# Build
forge build
```

## Deployment

```bash
# Set environment
export WHITELISTED_TOKENS=0xYourTokenAddress
export CAST_ACCOUNT=your-keystore-account
export SENDER_ADDRESS=0xYourAddress

# Deploy (dry run)
forge script script/DeployMPPEscrow.s.sol --rpc-url $RPC_URL --sender $SENDER_ADDRESS

# Deploy (broadcast)
forge script script/DeployMPPEscrow.s.sol --rpc-url $RPC_URL --broadcast --account $CAST_ACCOUNT --sender $SENDER_ADDRESS
```

See the [Foundry docs on cast wallet](https://book.getfoundry.sh/reference/cast/cast-wallet) for keystore setup. Do not use raw private key env vars.
